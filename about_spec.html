ABOUT File Specification v0.6.1

Purpose

The purpose of an ABOUT file is to provide a simple way to document the provenance (origin and license) and other important or interesting information about a software component. An ABOUT file is a small text file stored in the codebase side-by-side with the software component that it documents. No modification of the documented software is needed. The format is plain text with name/value pairs separated by a colon and simple conventions. It is easy to read and create by hand and is designed first for humans, rather than machines. The format is well-defined and structured just enough to make it easy to process with software as well. It contains enough information to fulfill key license requirements such as creating credits or attribution notices, collecting redistributable source code, or providing information about new versions of a software component.

Getting Started

A simple and valid ABOUT file named httpd-2.4.3.tar.gz.ABOUT may look like this: 

      name: Apache HTTP Server
      home_url: http://httpd.apache.org
      download_url: http://archive.apache.org/dist/httpd/httpd-2.4.3.tar.gz
      version: 2.4.3
      date: 2012-08-21
      license_spdx: Apache-2.0
      license_text_file: httpd.LICENSE
      copyright: Copyright 2012 The Apache Software Foundation.
      notice_file: httpd.NOTICE

The meaning of this file is:

  • This ABOUT file documents the file httpd-2.4.3.tar.gz, because the ABOUT file name is httpd-2.4.3.tar.gz.ABOUT. The file httpd-2.4.3.tar.gz.ABOUT is stored in the same folder as the documented file httpd-2.4.3.tar.gz.
  • The name of this component is "Apache HTTP Server" and its version is "2.4.3".
  • The home URL for this component is http://httpd.apache.org
  • The file httpd-2.4.3.tar.gz was originally downloaded from http://archive.apache.org/dist/httpd/httpd-2.4.3.tar.gz
  • In the same folder, there is a file named httpd.LICENSE that contains the license text for this component. and a file named httpd.NOTICE that contains the notice text for this component.

Specification

The format of an ABOUT file format is simple: lines of colon-separated name:value pairs. It is loosely based on the Email header field format as specified in RFC5322 at http://tools.ietf.org/html/rfc5322 . This provides a convenient and well-defined specification and there are many available tools and libraries that can parse and interpret this format. An ABOUT file content is encoded as UTF-8.

Field names syntax

Field names can contain only these US-ASCII characters:

  • digits from 0 to 9
  • uppercase and lowercase letters from A to Z and a to z
  • the "_" underscore sign.

    • The formal lexical definition is: 

          field_name ::=  (letter|"_") (letter | digit | "_")*
      letter     ::=  lowercase | uppercase
      lowercase  ::=  "a"..."z"
      uppercase  ::=  "A"..."Z"
      digit      ::=  "0"..."9"

    The case of a field name is not significant such that URL and url are the same field name.

    Fields are mandatory, optional, extensions or ignored.

    As defined in this specification, a field can be mandatory, optional or ignored. An ignored field is a field with a name that is not defined in this specification. Custom extensions are also supported and are processed by tools as ignored fields unless you have built a custom tool to understand and process these extensions.

    Field body

    The body of a field and its encoding are defined according to the RFC5322 Email header specification Section 2.2 http://tools.ietf.org/html/rfc5322#section-2.2

    For instance, a field body can contain multiple lines; space folding applies. In this example the description is one logical line: 

    description: This is a long description for a
  software component that spans multiple lines with arbitrary line breaks.

    Referencing the file or folder documented by an ABOUT file.

    An ABOUT file documents one file or folder. By convention, you store an ABOUT file in the same folder as the file or folder that it documents. To name an ABOUT file, simply append the ".ABOUT" extension to the name of the file or folder that it documents.

    For instance:

    • A file named linux-3.7-rc7.tar.bz2.ABOUT provides documentation about the linux-3.7-rc7.tar.bz2 file stored in the same folder.
    • A file named nodejs.ABOUT provides documentation about the nodejs folder stored side by side with the nodejs.ABOUT file.

    Alternatively an ABOUT file may contain the "about_file" field that contains a POSIX path to the file it documents. This path is relative to the path of the ABOUT file.

    For instance, a file named django.ABOUT may contain the following field to provide documentation about the django-1.2.3.tar.gz file stored in the same folder: 

          about_file: django-1.2.3.tar.gz

    The about_file field is optional and, if present, takes precedence over a file possibly pointed to by the ABOUT file name itself. This field can appear only once in the file.

    Fields validation

    When a tool processes an ABOUT file, it must produce a warning or error if a field is invalid. A field can be invalid for several reasons, such as an invalid syntax or an invalid content. A tool should report details on the nature of the error. The validation process should check that each field name:value pair is syntactically correct and that it contains correct values according to its concise, common sense definition in this specification. Certain fields may be used for additional specific validations (such as extensions for checksum verification, URL validity, path pointer to an existing text file, and so forth). Ignored fields should trigger a warning.

    The field order does not matter with the following exception: multiple occurrences of a field name are possible and must be interpreted such that only the last occurrence is considered as the value for this field name. Previous occurrences in the same file must be treated as ignored fields.

    A tool processing an ABOUT file must issue a warning when a field name occurs more than once in an ABOUT file (as for any other ignored field).

    Fields referencing files and URLs

    The body of a field may be contained in another file. In this case the field name is suffixed with "_file" and the field body must be a POSIX path to the file that contains the body of the field. This path must be relative to the path of the ABOUT file.

    For instance the full license text for a component is often stored in a separate file named COPYING: 

          license_text_file: COPYING

    The body of a field may reference URLs such as a homepage or a download. In this case the field name is suffixed with "_url" and the field body must be a valid absolute URL. For instance, a download URL is referenced this way: 

          download_url: http://www.kernel.org/pub/linux/kernel/v3.0/linux-3.4.20.tar.bz2

    Mandatory fields

    When a tool processes an ABOUT file, it must issue an error if a mandatory field is missing.

    • name: Component name.
    • version: Component version. A component usually has a version, such as a revision number or hash from a version control system (for a snapshot checked out from SCMs such as Subversion or Git). If not available, the version should be the date the component was provisioned, in a format such as 'YYYY-MM-DD'.

    Optional Information fields

    • about_format: The default value is 1.0. This defines the ABOUT file format specification and is provided as a hint to readers and tools in order to support future versions of this specification.
    • date: The date ('YYYY-MM-DD') when this ABOUT file was created or last validated (such as the date when the URLs were verified).
    • description: Component description.
    • home_url: URL to the homepage for this component.
    • download_url: A direct URL to download the original file documented by this ABOUT file.
    • readme: Readme (README) information for the component, usually a relatively short amount of text.
    • readme_file: The name of a readme (README) text file associated with the component.
    • install: Installation information for the component, usually a relatively short amount of text.
    • install_file: The name of an install (INSTALL) text file associated with the component.
    • changelog: Changelog text for the component, usually a relatively short amount of text.
    • changelog_file: The name of a changelog (CHANGELOG) text file associated with the component.
    • news: News text for the component, usually a relatively short amount of text.
    • news_file: The name of a news (NEWS) text file associated with the component.
    • news_url: A URL that provides an internet news feed for the component.
    • notes: Notes and comments about the component, in addition to those already provided in the description field.
    • usage: A concise description of the intended usage for the component. Suggested values include:
      • internal: Internal use only. Not part of the distributed software.
      • product: Deployed on the product. Used in the distributed software.
      • tool: Development/testing tool. Not part of the distributed software.

    Optional Author and Licensing fields

    • contact: Contact information (such as an email address or physical address) for the component provider.
    • organization: The name of the primary organization or persons that authored or provided the component.
    • copyright: Copyright statement for the component, usually a relatively short amount of text.
    • copyright_file: The name of a copyright (COPYRIGHT) file associated with the component.
    • notice: Legal notice for the component, usually a relatively short amount of text.
    • notice_file: The name of a legal notice (NOTICE) file associated with the component.
    • notice_url: URL to a legal notice for the component.
    • license_text: License text that applies to the component, usually for relatively simple license text.
    • license_text_file: The name of a license file (for example, a LICENSE or COPYING file in the component archive) that applies to the component.
    • license_url: URL to the license text for the component.
    • license_spdx: The SPDX license short form identifiers for the license of this component. See http://spdx.org/licenses/ for details. You can separate each identifier using " or " and " and " as defined in the SPDX specification 1.1 to document the relationship between multiple license identifiers, such as a choice of license.
    • redistribute_sources: yes or no. Defaults to no. Set this to yes if the component license requires code redistribution in source form.

    Optional Extension fields

    You can create extension fields by prefixing them with a short acronym to distinguish these from the standard fields. You should provide documentation for these extensions and create or extend existing tools to support these extensions. Other tools will ignore these extensions.

    Some basic extensions are provided here as well.

    Optional Extension fields to reference files stored in a version control system

    These fields provide a simple way to reference files stored in a version control system. There are a large number of version control tools such as CVS, Subversion, Git, ClearCase and GNU Arch. Accurate addressing of a file or folder revision in each version control tool in a uniform way may not be possible. Some tools may require access control via user/password or certificate; this information should not be stored in an ABOUT file. The suggested approach is to define the "scm_" extension prefix and a few common fields to handle the diversity of ways that SCM tools reference files:

    • scm_tool: SCM tool such as git, svn, cvs, etc.
    • scm_repository: Typically a URL or some other identifier used by the tool to point to a repository, folder or file, such as an SVN or Git repository URL.
    • scm_path: Path used by a particular SCM to point to a file, folder or module inside a repository.
    • scm_tag: tag name or path used by a particular SCM.
    • scm_branch: branch name or path used by a particular SCM.
    • scm_rev: revision identifier such as a revision hash or version number.

    Some scm_* examples: 

          scm_tool: svn
      scm_repository: http://svn.code.sf.net/p/inkscape/code/inkscape_project/trunk/inkscape_planet/
      scm_rev: 22886

    or 

          scm_tool: git
      scm_repository: git://git.kernel.org/pub/scm/linux/kernel/git/stable/linux-stable.git
      scm_path: tools/lib/traceevent
      scm_rev: b59958d90b3e75a3b66cd311661535f94f5be4d1

    Optional Extension fields for checksums and digital signature

    These fields support checksums (such as SHA1) and signatures (such as GPG) commonly provided with downloadable archives to verify their integrity. A tool could optionally use these to verify the integrity of a file documented by an ABOUT file.

    The approach is to define "checksum_" and "signature_" extension prefixes padded with an identifier for each type of signature or checksum such as "checksum_sha1".

    • signature_<tool id>: Signature for the documented for the file documented by this ABOUT file. You may use signature_<tool id>_file to reference a pre-existing file as these signature files for GPG and PGP are commonly provided for verifying code archives such as here: http://www.kernel.org/pub/linux/kernel/v3.0/linux-3.1.7.tar.sign
    • checksum_sha1, checksum_md5, checksum_sha256: Checksum for the file documented by this ABOUT file. The suffix defines the checksum algorithm type such as "_md5", "_sha1", "_sha256". The checksum algorithms and formats are as defined in the GNU Coreutils tools md5sum, sha1sum and sha256sum commands ( http://www.gnu.org/software/coreutils/ ) that follow the specifications defined in the FIPS-180-2 standard ( http://csrc.nist.gov/publications/fips/fips180-2/fips180-2.pdf ). Files can be referenced with the "_file" suffix such as with "checksum_sha1_file" to reference checksums files commonly provided such as here: http://mirrors.ibiblio.org/maven2/org/jdom/jdom/1.1.2/jdom-1.1.2-javadoc.jar.md5

    Some examples: 

          signature_gpg_file: linux-3.1.7.tar.sign
      checksum_sha1: 87aaf7bc7e8715f0455997bb8c6791aa

    Optional Extension fields for DejaCode

    These fields provide a reference to DejaCode Enterprise objects via URN.

    • dje_component: The DejaCode Enterprise URN for the component.
    • dje_license: The DejaCode Enterprise URNs of the licenses for the component.
    • dje_organization: The DejaCode Enterprise URN for the component organization.
