ABOUT File Specification v0.5.1

Purpose

The purpose of an ABOUT file is to provide a minimalist and simple way to document the provenance (origin and license) and other important or interesting information about a software component. An ABOUT file is a small text file stored in the codebase side-by-side with the software component it documents. No modification of the documented software is needed. The format is plain text with name/value pairs separated by a colon and simple conventions. It is easy to read and create by hand and is designed first for humans, not for machines. The format is well-defined and structured just enough to make it easy to process with software too. It contains enough information to fulfill key license requirements such as creating credits or attribution notices and collect redistributable source code or get informed about new versions of a software component.

Getting Started

A simple and valid ABOUT file named httpd-2.4.3.tar.gz.ABOUT may look like this: 

      name: Apache HTTP Server
      home_url: http://httpd.apache.org
      download_url: http://archive.apache.org/dist/httpd/httpd-2.4.3.tar.gz
      version: 2.4.3
      date: 2012-08-21
      license_spdx: Apache-2.0
      license_text_file: httpd.LICENSE
      copyright: Copyright 2012 The Apache Software Foundation.
      notice_file: httpd.NOTICE

The meaning of this file is:

  • This ABOUT file documents the file httpd-2.4.3.tar.gz, because the ABOUT file name is httpd-2.4.3.tar.gz.ABOUT. The file httpd-2.4.3.tar.gz.ABOUT is stored in the same folder as the documented file httpd-2.4.3.tar.gz.
  • The name of this component is "Apache HTTP Server" and it's version is "2.4.3".
  • The home URL for this component is "http://httpd.apache.org".
  • The file httpd-2.4.3.tar.gz was originally downloaded from http://archive.apache.org/dist/httpd/httpd-2.4.3.tar.gz
  • In the same folder, there is a file named httpd.LICENSE that contains the license text for this component. and a file named httpd.NOTICE that contains the notice text for this component.

Specification

The format of an ABOUT file format is simple: lines of colon-separated name:values pairs. It is loosely based on the Email header field format as specified in RFC5322 at http://tools.ietf.org/html/rfc5322 . This provides a convenient and well-defined specification and there are many available tools and libraries that can parse and interpret this format. An ABOUT file content is encoded as UTF-8.

Field names syntax

Field names can contain only these US-ASCII characters: digits from 0 to 9, uppercase and lowercase letters from A to Z and the "_" underscore sign. The formal lexical definition is: 

      field_name ::=  (letter|"_") (letter | digit | "_")*
      letter     ::=  lowercase | uppercase
      lowercase  ::=  "a"..."z"
      uppercase  ::=  "A"..."Z"
      digit      ::=  "0"..."9"

The case of a field name is not significant such that URL and url are the same field name.

Fields are mandatory, optional, extensions or ignored.

As defined in this specification, a field can be mandatory, optional or ignored. Ignored fields are fields with a name that is not defined in this specification. Custom extensions are also supported and must be processed by tools as ignored fields unless you have built a custom tool to understand and process these extensions.

Field body

The body of a field and its encoding are defined according to the RFC5322 Email header specification Section 2.2 http://tools.ietf.org/html/rfc5322#section-2.2

For instance, a field body can contain multiple lines and space folding applies. In this example the description is one logical line: 

description: This is a long description for a
  software component that spans multiple lines with arbitrary line breaks.

Referencing the file or folder documented by an ABOUT file.

An ABOUT file documents one file or folder. By convention, an ABOUT file is stored in the same folder as the file or folder it documents. An ABOUT file is named after the file or folder it documents adding a ".ABOUT" extension.

For instance:

  • a file named linux-3.7-rc7.tar.bz2.ABOUT provides documentation about the linux-3.7-rc7.tar.bz2 file stored in the same folder.
  • a file named nodejs.ABOUT provides documentation about the nodejs folder stored side by side with the nodejs.ABOUT file.

Alternatively an ABOUT file may contain the "about_file" field that contains a POSIX path to the file it documents. This path is relative to the path of the ABOUT file.

For instance, a file named django.ABOUT may contain the following field to provide documentation about the django-1.2.3.tar.gz file stored in the same folder: 

      about_file: django-1.2.3.tar.gz

The about_file field is optional and if present, it takes precedence over a file possibly pointed to by the ABOUT file name itself. This field can appear only once.

Fields validation

When a tool processes an ABOUT file it must produce a warning or error if a field is invalid. A field can be invalid for several reason such as an invalid syntax or an invalid content. Tools should report details on the nature of the error. The validation process should check that each field name and values are syntactically correct and contain correct values according to its (terse but common sense) definition in this specification. Certain fields may be used for additional specific validations (such as extensions for checksums verification, URL validity, path pointing to an existing text file, etc). Ignored fields should trigger a warning.

The field order does not matter with the following exception: multiple occurrences of a field name are possible and must be interpreted such that only the last occurrence is considered as the value for this field name. Previous occurrences must be treated as ignored fields.

Tools processing an ABOUT file must issue a warning when a field name occurs more than once in an ABOUT file (as for any other ignored field).

Fields referencing files and URLs

The body of a field may be contained in another file. In this case the field name is suffixed with "_file" and the field body must be a POSIX path to the file that contains the body of the field. This path must be relative to the path of the ABOUT file.

For instance the full license text for a component is often stored in a separate file named COPYING: 

      license_text_file: COPYING

The body of a field may reference URLs such as a homepage or a download. In this case the field name is suffixed with "_url" and the field body must be a valid absolute URL. For instance the download URL is referenced this way: 

      download_url: http://www.kernel.org/pub/linux/kernel/v3.0/linux-3.4.20.tar.bz2

Fields referencing files inside archives

To document files stored in an existing archive without needing to modify this archive, file-related fields can point to files stored inside an archive. (Suffixed fields with "_file"). The field body is a POSIX path to the archive file and the path to the file inside the archive. To address the file inside an archive, use the path as listed by the extracting tool commonly used to extract this archive type. This type of reference can be used for any "_file"-suffixed field, including the "about_file" field.

For instance compressed Tar archives are commonly extracted with GNU Tar and the archive file listing is obtained this way: 

      $ tar -tzf httpd-2.4.3.tar.gz
      httpd-2.4.3/
      httpd-2.4.3/CHANGES
      httpd-2.4.3/httpd.dsp
      httpd-2.4.3/LICENSE
      httpd-2.4.3/Makefile.in
      httpd-2.4.3/NOTICE
      httpd-2.4.3/README
      .....

Here the ABOUT file httpd-2.4.3.tar.gz.ABOUT could contain these fields to reference the existing LICENSE and NOTICE files provided in the archive: 

      license_text_file: httpd-2.4.3.tar.gz/httpd-2.4.3/LICENSE
      notice_file: httpd-2.4.3.tar.gz/httpd-2.4.3/NOTICE

Mandatory fields

When a tool processes an ABOUT file it must issue an error if these fields are missing.

  • name: The name for the component
  • version: The component version. Most components have a version. A version identifier can be a revision or hash from a version control system (for a snapshot checked out from SCMs such as Subversion or Git). If not available, the version should be the date the component was provisioned in ISO-like format as 'YYYY-MM-DD'.

Optional Information fields

  • about_format: This defines the ABOUT file format specification used and is provided as hint to readers and tools in order to support future versions of this specification. Default value is 1.0 if omitted.
  • date: the date when this ABOUT file was created or when the fields values were valid (for instance the date when URLs were valid)
  • description: A description for this component.
  • home_url: URL to the homepage for this component.
  • download_url: A direct URL to download the original file documented by this ABOUT file.
  • readme: Readme information for this component. You may use readme_file when this is a long text or a pre-existing file as README files are commonly found in code archives.
  • install: Installation information for this component. You may use install_file when this is a long text or a pre-existing file as INSTALL files are commonly found in code archives.
  • changelog: Changelog text for this component. You may use changelog_file when this is a long text or a pre-existing file as CHANGELOG files are commonly found in code archives.
  • news: News text for this component. You may use news_file when this is a long text or a pre-existing file as NEWS files are commonly found in code archives. You may use news_url to point to an internet news feed for this component.
  • notes: Your notes and comments about this component.
  • usage: Describe the intended usage for this component. Possible values are limited to "internal, product and tool" each with the following definitions:
    • internal: used only internally, not part of the distributed software that uses this component.
    • product: used in the distributed software.
    • tool: tool used for development, not part of the distributed software.

Optional authors and licensing fields

  • contact: Contact information (such as an email address or physical address) for the component providers.
  • organization: The name of the primary organizations or persons that authored or provided this component.
  • copyright: Copyright statement for this component. You may use copyright_file when this is a long text or to reference a pre-existing file. For instance COPYRIGHT files are commonly found in code archives.
  • notice: Text containing a legal notice for this component. You may use notice_file when this is a long text or to reference a pre-existing file. For instance NOTICE files are commonly found in code archives.
  • notice_url: URL to a legal notice for this component.
  • license_text: License text for this component. You may use license_text_file when this is a long text or to reference a pre-existing file. For instance LICENSE or COPYING files are commonly found in code archives.
  • license_url: URL to the license text for this component.
  • license_spdx: The SPDX license short form identifiers for the license of this component. See http://spdx.org/licenses/ for details. You can separate each identifier with an " or " and " and " as defined in the SPDX specification 1.1 to document the relationship between multiple license identifiers, such as a choice of license.
  • redistribute_sources: yes or no. Defaults to no if not present. Describes if this component license requires redistribution in source form.

Optional Extension fields

You can create extension fields by prefixing them with a short acronym to distinguish these from the standard fields. You should provide documentation for these extensions and create or extend existing tools to support these extensions. Other tools will ignore these extensions.

Some basic extensions are provided here too.

Optional Extension fields to reference files stored in version control system

These fields provide a simple way to reference files stored in versions control system. There is a large number of version control tools such as CVS, Subversion, Git, ClearCase or GNU Arch and addressing accurately a file or folder revision in each of these tool in a uniform way is not an easy task. Some tool may require access control with user/password or certificates and typically this information should not be stored in the ABOUT files. The proposed approach is to define the "scm_" extension prefix and a few common fields to handle the diversity of SCM tools ways to reference files:

  • scm_tool: SCM tool such as git, svn, cvs, etc.
  • scm_repository: Typically a URL or some other identifier used by the tool to point to a repository, folder or file, such as an SVN or Git repository URL.
  • scm_path: Path to a file, folder or module used by certain SCM pointing inside a repository.
  • scm_tag: tag name or path used by certain SCM.
  • scm_branch: branch name or path used by certain SCM.
  • scm_rev: revision identifier such as a revision hash or version number.

Some examples: 

      scm_tool: svn
      scm_repository: http://svn.code.sf.net/p/inkscape/code/inkscape_project/trunk/inkscape_planet/
      scm_rev: 22886

or 

      scm_tool: git
      scm_repository: git://git.kernel.org/pub/scm/linux/kernel/git/stable/linux-stable.git
      scm_path: tools/lib/traceevent
      scm_rev: b59958d90b3e75a3b66cd311661535f94f5be4d1

Optional Extension fields for checksums and digital signature

These fields support checksums (such as SHA1) and signatures (such as GPG) commonly provided with downloadable archives to verify their integrity. Tools could optionally use these to verify the integrity of a file documented by an ABOUT file.

The approach is to define the "checksum_" and "signature_" extensions prefix padded with an identifier for each type of signature or checksum such as in "scm_sha1".

  • signature_<tool id>: Signature for the documented for the file documented by this ABOUT file. You may use signature_<tool id>_file to reference a pre-existing file as these signature files for GPG and PGP are commonly provided for verifying code archives such as here: http://www.kernel.org/pub/linux/kernel/v3.0/linux-3.1.7.tar.sign
  • checksum_sha1, checksum_md5, checksum_sha256: Checksum for the file documented by this ABOUT file. The suffix defines the checksum algorithm type such as "_md5", "_sha1", "_sha256". The checksum algorithms and formats are as defined in the GNU Coreutils tools md5sum, sha1sum and sha256sum commands ( http://www.gnu.org/software/coreutils/ ) that follow the specifications defined in the FIPS-180-2 standard ( http://csrc.nist.gov/publications/fips/fips180-2/fips180-2.pdf ). Files can be referenced with the "_file" suffix such as with "checksum_sha1_file" to reference checksums files commonly provided such as here: http://mirrors.ibiblio.org/maven2/org/jdom/jdom/1.1.2/jdom-1.1.2-javadoc.jar.md5

Some examples: 

      signature_gpg_file: linux-3.1.7.tar.sign
      checksum_sha1: 87aaf7bc7e8715f0455997bb8c6791aa

Optional Extension fields for DejaCode

These fields allow referencing DejaCode Enterprise objects with their URNs.

  • dje_component: The DejaCode Enterprise URN for this component.
  • dje_license: The DejaCode Enterprise URNs of the licenses for this component.
  • dje_organization: The DejaCode Enterprise URN for this component organization.
